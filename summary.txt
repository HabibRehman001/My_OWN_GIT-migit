1. WHAT IS MIGIT?
--------------------------------------------------------------------------------

migit is a TypeScript command-line application that implements a simplified
Git-like version control system (VCS). It is NOT a full Git replacement — it is
a learning / prototype VCS with these goals:

  • Track file changes using content-addressable storage (like Git objects)
  • Stage files, commit snapshots, view history, manage branches
  • Optional smart commit messages via Google Gemini API
  • Auto-scan a codebase and produce markdown documentation
  • Command history logging with optional AI explanation
  • Repository health checks (doctor)

Tech stack:
  • Language: TypeScript (ES modules, compiled to dist/)
  • CLI framework: Commander.js
  • Runtime: Node.js (ES2022)
  • AI (optional): Google Gemini via GEMINI_API_KEY environment variable
  • No database — everything stored as files inside `.migit/`


2. HOW TO RUN (GLOBAL CLI)
--------------------------------------------------------------------------------

Primary usage (after linking):

  npm install
  npm run build          # tsc + chmod +x dist/cli.js
  npm link               # installs `migit` on your PATH
  migit init
  migit --help

package.json bin entry:

  "bin": { "migit": "./dist/cli.js" }

src/cli.ts starts with:

  #!/usr/bin/env node

TypeScript preserves the shebang in dist/cli.js. npm link creates a global
shim pointing at the built CLI.

Development without linking:

  npm start -- init
  npm start -- status


3. HOW THE WHOLE SYSTEM WORKS (HIGH-LEVEL FLOW)
--------------------------------------------------------------------------------

When you run `migit <command>`:

  STEP 1 — ENTRY (src/cli.ts)
    loadEnv() loads .env from cwd, repo root, or package dir.
    Commander.js parses argv and dispatches to a subcommand handler.

  STEP 2 — HISTORY WRAPPER (src/cli/with-history.ts)
    Every command runs through withHistoryAction() → runWithHistory().
    On success or failure, a structured JSON line is appended to
    .migit/history.log (API keys and secrets are redacted from args).

  STEP 3 — COMMAND LAYER (src/commands/*.command.ts)
    Thin wrappers. Most use Repository.open() (walks up to find .migit).
    init uses Repository.at(cwd) — never walks up to a parent repo.

  STEP 4 — REPOSITORY DISCOVERY (src/utils/paths.ts → findRepositoryRoot)
    Walks up from cwd until `<dir>/.migit` exists; that dir is the repo root.
    Example: cwd = project/src/controllers → finds project/.migit → root = project/

  STEP 5 — CORE LAYER (src/core/)
    Repository orchestrates ObjectStore, IndexStore, Refs, Scanner, ConfigStore.

  STEP 6 — ON DISK
    All metadata and objects live in `.migit/` at the repo root.


4. THE `.migit/` REPOSITORY STRUCTURE ON DISK
--------------------------------------------------------------------------------

  your-project/
  ├── .migitignore          ← ignore patterns (created on init if missing)
  ├── .migit/
  │   ├── HEAD              ← "ref: refs/heads/main" (active branch)
  │   ├── config.json       ← author + AI provider (never stores API keys)
  │   ├── index             ← JSON array of staged/tracked files
  │   ├── history.log       ← JSONL command audit log
  │   ├── objects/          ← content-addressed blobs, trees, and commits (gzip)
  │   │   └── ab/
  │   │       └── cdef1234…
  │   └── refs/
  │       └── heads/
  │           ├── main
  │           └── feature-x
  └── your source files…

HEAD vs branch refs (critical):

  .migit/HEAD                    → which branch is checked OUT
  .migit/refs/heads/main         → latest commit ON main
  .migit/refs/heads/feature-x    → latest commit ON feature-x

  migit commit     → updates refs/heads/<current-branch> only
  migit checkout X → updates HEAD only; restores working tree + index from X

Object format on disk:

  Stored as gzip( "blob <size>\0<content>" )
         or gzip( "tree <size>\0<json-map>" )
         or gzip( "commit <size>\0<json>" )
  Object ID = SHA-256 of the full uncompressed header+payload string.

  Tree payload: JSON object mapping repo-relative paths → blob hashes.
  Commits reference tree hashes; trees reference blob hashes.

config.json example (safe — no API keys):

  {
    "user": { "name": "...", "email": "..." },
    "ai": { "provider": "gemini" }
  }

  Optional: "model" in ai section or MIGIT_AI_MODEL env var.
  API key: GEMINI_API_KEY in .env only — never in config.json.

history.log entry format (one JSON object per line):

  {
    "command": "commit",
    "args": ["commit", "--smart"],
    "status": "success",
    "durationMs": 1234,
    "timestamp": "2026-06-26T12:00:00.000Z"
  }


5. ALL CLI COMMANDS (WHAT THEY DO & HOW)
--------------------------------------------------------------------------------

5.1  migit init [-n name] [-e email]
  • Repository.at(process.cwd()) — never discovers parent repos
  • Creates .migit/, objects/, refs/heads/, HEAD → refs/heads/main
  • Saves empty index [], writes config.json with author defaults
  • Creates .migitignore with default patterns if missing
  • Does NOT create an initial commit

5.2  migit add <paths...>
  • Rejects paths outside repo root (../ escapes blocked)
  • Scanner walks paths; respects .migitignore + built-in ignores
  • Each file: read bytes → objectStore.writeBlob() → hash stored in index
  • indexStore.merge() overlays updates — never wipes entire index
  • Staged deletion: files removed from disk within add scope are removed from index
  • Skips: symlinks, files > 10 MB, permission-denied (with warnings)
  • Binary files stored as blobs normally

5.3  migit status
  • Three-way compare: HEAD ↔ Index ↔ Working directory
  • status-engine.ts loads HEAD tree, index, scans working tree
  • Output per path: staged state + working state (or omitted if clean)
  • Staged: added / modified / deleted (HEAD ≠ index)
  • Working: modified / deleted / untracked (index ≠ disk)

5.4  migit commit -m "message"
  • Builds tree object from complete index via objectStore.writeTree()
  • Creates commit object with author from config.json
  • Updates current branch ref to new commit hash
  • Index is NOT cleared — stays equal to HEAD after commit

5.5  migit commit --smart [--paths-only] [-y]
  • Builds structured change summary from staged files (HEAD vs index diffs)
  • Default mode: sends metadata + redacted diff samples to Gemini (not full files)
  • --paths-only: sends only file paths (legacy/simple mode)
  • Excludes .env and sensitive files from analysis
  • Gemini: 15s timeout, 1 retry, 72-char message limit, response validation
  • Falls back to deterministic message if AI fails
  • Prompts user to confirm message (skip with -y)
  • Never sends API keys or .env contents to Gemini

5.6  migit log [-n count]
  • Walks parent chain from HEAD commit (default 10 commits)
  • Prints hash, author, date, message

5.7  migit branch [name] [-d]
  • No name: list branches, * marks current
  • With name: create branch at current HEAD
  • With -d: delete branch ref file
  • Safety rules (see section 10)

5.8  migit checkout <branch> [-f|--force]
  • Validates branch exists and has commits
  • Dirty-tree protection (default): blocks if staged changes, unstaged changes
    on tracked files, or untracked files that would be overwritten by target branch
  • Lists conflicting file paths in error message
  • --force skips protection and switches anyway
  • Restores working tree from target commit tree, replaces index, updates HEAD

5.9  migit config [key] [value] [--list]
  • Get/set .migit/config.json values
  • Keys: user.name, user.email, ai.provider, ai.model
  • Rejects ai.key / apiKey / token keys — use GEMINI_API_KEY env instead
  • --list prints full config JSON

5.10 migit history [--explain]
  • Shows .migit/history.log entries (command, args, status, duration)
  • migit history explain       → plain-text summary of recent commands
  • migit history explain --ai  → Gemini summary
  • migit history --explain     → alias for explain --ai

5.11 migit document [-o path] [--force]
  • Runs analyzers in parallel → DOCUMENTATION.md (or custom path)
  • Refuses to overwrite an existing output file unless --force is passed
  • Respects .migitignore; never includes .env or credential files
  • Output via atomicWrite()

5.12 migit doctor
  • Deep integrity checks (see section 9)
  • Exit code 1 if any issues found


6. IGNORE RULES (.migitignore)
--------------------------------------------------------------------------------

File: .migitignore at repository root (auto-created on init).

Example patterns (gitignore-like subset):
  node_modules/
  dist/
  build/
  coverage/
  .env
  .env.*
  *.log
  uploads/

Always ignored (cannot be overridden):
  .migit/   .git/   node_modules/   .env   .env.*

Implementation:
  src/utils/ignore-rules.ts  — loads .migitignore, parses patterns, caches rules
  src/utils/sensitive-files.ts — credential file detection
  src/utils/ignore.ts        — shouldIgnorePath() async helper

Walk policy (scanner, status, document):
  • Reject paths outside repository root (path-guard.ts)
  • Do not follow symbolic links
  • Skip files > 10 MB (MAX_STAGE_FILE_BYTES)
  • Skip permission-denied paths with warning
  • Binary files: stored as blobs; smart-commit skips content analysis for binary

8. COMMAND HISTORY LOGGING
--------------------------------------------------------------------------------

Every subcommand is wrapped with withHistoryAction() → runWithHistory().

Flow:
  1. Record start time
  2. Run command handler
  3. Append JSON line to .migit/history.log (success or failed)
  4. Args sanitized via history-sanitize.ts (redacts keys, tokens, passwords)
  5. Errors redacted via redact-secrets.ts before persistence

Special case: migit init logs after .migit/ exists (post-operation append).

Files:

  src/history/history-logger.ts     — atomicAppendLine to history.log
  src/history/history-reader.ts     — reads JSONL back
  src/history/history-format.ts     — display + explain formatting
  src/history/history-sanitize.ts   — argv + error redaction
  src/cli/with-history.ts           — Commander action adapter


9. SMART COMMIT (AI) — FULL PIPELINE
--------------------------------------------------------------------------------

Trigger: migit commit --smart

Steps:
  1. buildStagedChangeSummary() — compares HEAD vs index for each staged path
  2. For each file: line diff stats, heuristic summary bullets, redacted diff sample
  3. Skips sensitive files (.env, credentials) entirely
  4. Builds JSON payload (not raw source) and sends to Gemini
  5. gemini-client.ts: x-goog-api-key header (not in URL), 15s timeout, 1 retry
  6. normalizeCommitMessage(): single line, max 72 chars, strip quotes/fences
  7. fallbackMessage() if AI unavailable or response invalid
  8. confirmCommitMessage() prompts user (unless -y)

Files:
  src/ai/staged-changes.ts    — HEAD↔index diff builder
  src/ai/change-summary.ts    — ChangeSummary types + JSON payload
  src/ai/diff-redactor.ts     — line diff, redaction, summary heuristics
  src/ai/smart-commit.ts      — orchestrates prompt + fallback
  src/ai/commit-message.ts    — validation + truncation
  src/ai/commit-fallback.ts   — deterministic messages
  src/ai/gemini-client.ts     — HTTP client with timeout/retry/redaction
  src/utils/prompt-confirm.ts — interactive y/N confirmation


10. CHECKOUT & BRANCH SAFETY
--------------------------------------------------------------------------------

Checkout guard (src/core/checkout-guard.ts):
  Blocks checkout when:
    • Any staged change (HEAD ≠ index)
    • Any unstaged change on tracked files
    • Untracked file that exists on target branch (would be overwritten)
  Allows checkout when untracked file is NOT on target branch.
  --force / -f skips all checks.

Branch name validation (src/utils/branch-name.ts):
  Rejects: ../main, feature/auth?, feature auth, empty, .lock suffix, etc.
  Allows: main, feature/auth, bugfix-123

Create branch:
  Rejects if branch name already exists.
  Rejects invalid names.

Delete branch:
  Rejects if branch does not exist.
  Rejects if branch is currently checked out:
    Cannot delete branch "main" because it is currently checked out.


11. DOCTOR / INTEGRITY CHECKER
--------------------------------------------------------------------------------

IntegrityChecker (src/core/integrity-checker.ts) verifies:

  • config.json exists; no forbidden secret fields
  • .migit/HEAD exists and has valid format
  • HEAD points to an existing branch ref
  • Every branch ref points to valid 64-char commit hash
  • Every object: decompress, validate header type/size, recompute SHA-256,
    compare hash to storage path (shard + filename)
  • Commit graph: each commit has valid tree object; parent chain walkable; no cycles
  • Each tree object references existing blob objects (doctor rejects blob-as-tree)
  • Index JSON valid; every index entry references existing blob
  • No leftover .tmp metadata files in .migit/

object-store.ts verifyStorage() is the foundation for object hash validation.

19. DATA FLOW DIAGRAMS (TEXT)
--------------------------------------------------------------------------------

INIT:
  migit init → Repository.at(cwd).init()
    → create .migit/ structure, HEAD, empty index, config.json, .migitignore
    → history logged after success

ADD:
  migit add paths → resolvePathInRepository → Scanner.stage()
    → writeBlob per file → indexStore.merge()
    → history logged

COMMIT (manual):
  migit commit -m "msg" → createSnapshot(tree) → createCommit() → refs.setHead()
    → author from config.json → index unchanged

COMMIT (smart):
  migit commit --smart → buildStagedChangeSummary → Gemini prompt
    → validate/fallback → user confirm → repo.commit()

CHECKOUT:
  migit checkout branch → findCheckoutBlockers (unless --force)
    → readBranch → readCommit → readTree → restoreTree → save index → setCurrentBranch

HISTORY:
  any command → runWithHistory → handler → appendHistory (sanitized args)

DOCTOR:
  migit doctor → IntegrityChecker.run() → list all issues → exit 1 if any

22. CURRENT LIMITATIONS
--------------------------------------------------------------------------------

  • Not full Git — no merge, rebase, stash, remotes, tags, submodules
  • No three-way merge or conflict markers
  • No CLI subprocess tests or Gemini integration tests (unit tests only)
  • Express/React/Mongoose analyzers use simple regex (not AST parsing)
  • Single author per commit (no co-authors)
  • No interactive staging (add -p) or partial hunk staging
  • Branch names validated but not as strictly as Git's full ref rules


23. SUMMARY IN ONE PARAGRAPH
--------------------------------------------------------------------------------

migit is a Node.js/TypeScript CLI that implements a lightweight Git-like VCS
using a `.migit/` folder with content-addressed gzip objects, a JSON index,
branch refs, and a real HEAD file. Repository discovery walks up from any
subfolder to find `.migit`. Commander.js routes twelve subcommands — all wrapped
in history logging with secret redaction — through Repository, which coordinates
staging (with .migitignore), three-way status, atomic metadata writes, safe
checkout with dirty-tree protection, branch safety rules, and deep integrity
checks via doctor. Optional Gemini powers smart commits (structured diff metadata,
